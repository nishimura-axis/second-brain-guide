## 背景・経緯

### エンターキーとは

- **エンターキー**（社名）から、Meta広告の支払い用に**クレジットカードを借用**している
- 借用したカードは、**TCB案件を運用している広告アカウントの支払い方法**に紐づけている

### なぜこの仕組みがあるのか（TCBの財務リスク回避）

| 項目 | 内容 |
|------|------|
| **対象案件** | TCB（東京中央美容外科） |
| **財務リスク** | TCBは財務的に懸念があった |
| **対策** | 消化金額の一時支払いを**ASPに持ってもらう**ことで、万が一の倒産等で売上が支払えなくなった場合にAXISが被害を被らないようにしている |
| **結果** | AXIS側でMeta広告の消化金額を集計し、エンターキーが出すカード明細の合計金額と**一致するかどうかの確認作業**が必要になった |

### 要するに

```
AXIS → エンターキーのカードでMeta広告費を支払う
         ↓
エンターキー → カード明細を出す
         ↓
AXIS（アド管理部） → Meta請求書PDFから消化金額を集計
         ↓
集計額とカード明細の合計が一致するか確認
```

---

## 業務フロー

### 現行フロー（GAS版）

1. Google DriveにMeta請求書PDF（月分）を格納
2. スプレッドシートからGASを実行
3. GASがPDFをOCRで読み取り、以下を抽出：
   - ファイル名、日付、取引ID、支払い方法、参照番号、支払金額
4. カード（支払い方法）別に合計金額を集計
5. エンターキーのカード明細と突合確認

### 課題

- GASの実行時間制限（無料6分/Workspace30分）により大量PDFの一括処理ができない
- 分割実行（200件ずつ）が必要で手間がかかる

### 移行先（GCP版 — 構築中）

- GCP Cloud Run に移行して実行時間制限を解消
- 詳細は [[Meta_Invoice_Reconciliation_GCP移行]] を参照
