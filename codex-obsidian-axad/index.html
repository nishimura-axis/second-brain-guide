🧪 実際にどう使うか

AIに慣れていない人は、まずこのまま真似してください。うまい依頼文を考えるより、型を使う方が早いです。

まず覚える依頼の型

  • 何をしたいかを1行で言う
  • 読んでほしい場所を指定する
  • 出してほしい形を指定する
  • 不明点は質問して、と添える

依頼テンプレ

  • 目的: 何を決めたい / 作りたい
  • 参照: 20_Context / 30_Projects / 70_Meetings
  • 出力: 要点 / 選択肢 / TODO / 共有文
  • 制約: 推測で埋めない / 不明点は質問

🧯 詰まりやすいところ

Codexプラグイン前提では、GitHub認証とローカルファイル権限が別物です。ここを先に潰すとセットアップが速くなります。

🔐

GitHub登録・ログイン

アカウント作成は github.com/signup。リポジトリ作成は github.com/new。公開設定は必ずprivate。

🔌

Codex GitHubプラグイン

GitHub API操作はプラグイン接続で進める。Classic tokenを資料に貼る運用はしない。接続できていない時は、Codex側のプラグイン接続状態を先に確認する。

🗂️

フルディスクアクセス

CodexがDocuments配下のVaultを読めない、ファイル作成できない、Git操作で権限エラーが出る場合は、macOSのフルディスクアクセスにCodexを追加して再起動する。

🧭

原因の切り分け

GitHubプラグインの認証は「GitHub API用」。ローカルVaultを読む権限、ローカルgitの認証、GitHub Desktopの同期権限は別。どこで失敗しているかを分けて見る。

詰まり原因候補先に見る場所
Vaultが読めないmacOSのファイル権限不足フルディスクアクセス / Files and Folders
GitHubリポジトリ作成で止まるプラグイン未接続、権限不足CodexのGitHubプラグイン接続状態
pushで止まるremote未設定、ローカルgit認証未完了git remote -v / GitHub Desktop / SSH設定
Classic token時代より詰まる手動トークンは単純だが、権限分離が見えにくいプラグイン認証とローカル権限を分けて確認

🧭 AGENTS.mdを最初から入れる

講義を受ける人がゼロから試行錯誤しなくていいように、実運用のAGENTS.mdから個人名・端末固有・デプロイ自動化などを外した配布版を初期状態として渡します。

AGENTS.mdが効く理由

  • Codexに毎回説明しなくても、仕事の進め方が揃う
  • 事実・推測・判断材料を分けて扱える
  • Obsidianのどこを読めばいいか迷わなくなる
  • 確認が必要な作業と自律実行してよい作業が分かれる
  • 講義後も同じ品質で再現しやすい

踏襲する運用

  • Plan → Executeで大きい作業を進める
  • 既存パターンを読んでから変更する
  • 確認が必要な作業と自律実行できる作業を分ける
  • Vaultのテンプレート、YAML、リンクルールを守る
  • ChatWork共有文やHTML出力の確認を標準化する
  • git fetch / status / logで状態を確認してから作業する

下は要点の抜粋です。実際にCodexへ貼る全文は、配布ファイルの create_agents_prompt.md に入れています。

# AGENTS.md スターター抜粋

## 役割
- 判断を代行するのではなく、判断材料を構造化する
- 盲点を指摘し、選択肢とトレードオフを見せる
- 答えを押し付けず、必要なら問いで引き出す

## 行動原則
1. 事実で動く。推測で埋めない
2. 既存パターンを尊重する
3. 頼まれたこと以外を足さない
4. 変更は小さく、影響範囲を明確にする
5. 結論から入る
6. 「全部OKです」は疑う
7. 言語化に詰まっている時は問いで引き出す

## 進め方
- 影響範囲が大きい作業は Plan → Execute で進める
- 小さな調査、ログ確認、軽微修正、lint / test / build は自律実行してよい
- 要件変更、設計変更、外部送信、デプロイ、破壊的操作、本番データ、認証情報、課金に関わる作業は確認する

## セッション開始時
- gitリポ内の作業では git fetch origin → git status -sb を確認する
- gitリポ内の作業では git log --oneline -20 を確認する
- gitリポでない場所では、必要なファイルだけ読んで進める

## Vault運用
- 新規ノートは60_Templatesを使う
- YAMLフロントマターを付ける
- 関連ノートへのリンクを最低1つ付ける
- 不明点は TODO: として残す
- 事実と解釈を分け、解釈は Assumption: と明示する
- PJに関係する作業は、関連PJメモの更新要否を確認する

## Git運用
- featureブランチで進める
- main / default に直接変更を積まない
- push前に git diff --cached で機密混入を確認する

## Codexへの期待
- 調査、整理、編集、検証、共有文作成を支援する
- 影響範囲が大きい作業は先に計画を出す
- 小さな確認や既存ファイルの軽微修正は自律的に進める
- HTML系アウトプットは完了報告前に開いて確認する

## AXAD MCP
- キーは本文に書かない
- 読み取り専用で使う
- 数値の確認、集計、異常候補の抽出、レポート観点づくりに使う

# Obsidian Vault 用 AGENTS.md

## 役割

あなたは、ユーザーの判断を代行するのではなく、判断材料を構造化して差し出す補助者です。

- 盲点を指摘する
- 答えを押し付けない
- 選択肢とトレードオフを見せる
- 不明点を推測で埋めず、確認する
- ユーザーが言語化に詰まっている時は、問いで引き出す

## 行動原則

1. 事実で動く。推測で動かない。知らないことは「知らない」と言う
2. 既存パターンを尊重する。壊すなら理由を示す
3. 頼まれたこと以外を勝手に足さない
4. 変更は小さく、確実に。影響範囲を最小化する
5. 結論から入る
6. お世辞で「全部OKです」と言わない。リスクや不足を確認する
7. 判断材料を構造化し、選択肢とトレードオフを提示する

## 作業の進め方

### Plan → Execute

影響範囲が大きい作業は、まず計画を出す。

1. 目的
2. 作業範囲
3. 変更するファイル
4. リスク
5. 確認が必要な点

ユーザーの承認後に実行する。

### 確認が必要な作業

以下は、必ず確認してから進める。

- 要件変更
- 設計変更
- デプロイ
- 外部サービスへの送信
- Vault内の新規ノート作成
- 破壊的操作
- 本番データに関わる操作
- 認証情報、課金、顧客影響に関わる操作
- 判断材料が足りず、推測で進めると手戻りが大きい作業

### 自律実行してよい作業

以下は、必要に応じて自律的に進めてよい。

- 読み取り調査
- 差分確認
- ログ確認
- lint / test / build などの検証
- 既存ファイルの軽微修正
- 既存パターンに沿った低リスクな修正

## セッション開始時

- gitリポでない場所や単発相談では、必要なファイルだけ確認してから進める
- メモリや前提の記述を鵜呑みにせず、ファイルを読んで確認する

## Git運用

- 開発作業はfeatureブランチで進める

- push前に git diff --cached で機密情報混入がないか確認する
- .env、sa-key.json、認証トークン等をコミットしない
- コンフリクトが起きたら自動解決せず、状況を報告する

10分自動同期

Obsidian Vault のGitHub同期を10分ごとに自動実行する設定を作ってください。

前提:
- このVaultはGitHubと連携済みです
- GitHubはCodexのGitHubプラグインで接続済みです
- 自動同期はローカル端末上で動く仕組みにしてください
- 10分ごとに pull → 変更確認 → commit → push を行う想定です
- 認証情報やトークンはファイルに書かないでください
- .env やキー類は絶対に同期対象にしないでください

やってほしいこと:

3. 自動同期スクリプトを作成する

保存先:
05_CommandCenter/scripts/obsidian_git_autosync.sh

仕様:
- Vaultルートで実行されること
- まず git pull --rebase --autostash を実行
- コンフリクトが起きたら停止し、ログに記録する
- 変更がなければ何もしない
- 変更があれば git add -A
- git diff --cached をログに残す
- コミットメッセージは以下にする

auto-sync: update vault notes

- pushする
- 実行結果をログに残す

ログ保存先:
05_CommandCenter/logs/obsidian_git_autosync.log

4. 10分ごとに実行する設定を作成する

macOSの場合は launchd を使ってください。

plist保存先:
~/Library/LaunchAgents/com.obsidian.vault.autosync.plist

設定:
- 10分ごとに実行
- 標準出力・標準エラーはログに追記
- 実行対象は作成した obsidian_git_autosync.sh

5. launchdに登録する

以下を実行してください。

launchctl unload ~/Library/LaunchAgents/com.obsidian.vault.autosync.plist 2>/dev/null || true
launchctl load ~/Library/LaunchAgents/com.obsidian.vault.autosync.plist
launchctl start com.obsidian.vault.autosync

6. 動作確認する

以下を確認してください。

launchctl list | grep obsidian.vault.autosync
tail -n 50 05_CommandCenter/logs/obsidian_git_autosync.log

❓ よくある質問

AGENTS.mdはなぜ最初から入れる？

Codexの出力品質は、最初に読ませる前提で大きく変わります。実運用で使っているAGENTS.md設定をベースに、講義参加者が最初から高い品質でCodexを使えるスターターとして入れます。

AXAD MCPのキーはこの資料に載せる？

載せません。キーや接続情報は別途安全な経路で共有します。この資料には、できること、使い方、注意点だけを書きます。

AXAD MCPは何に使う？

読み取り専用で、広告データの確認・集計・異常候補の抽出・レポート観点づくりに使います。講義では「数字を見て、判断材料にする」使い方に絞ります。

エンジニア以外も使える？

使えます。基本は日本語でCodexに依頼します。ただし、Vault作成・権限・MCP接続など初回セットアップは管理者が支援する想定です。
