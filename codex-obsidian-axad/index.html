🧠 なぜ「第二の脳」を作るのか

このセットアップは、ツールを入れること自体が目的ではありません。仕事の前提、判断、学びをObsidianに残し、Codexがそれを読んで作業できる状態を作るためのものです。

📥

頭の中を外に出す

毎回思い出して説明していた前提、顧客理解、広告運用の判断軸、会議で決まったことをObsidianに置きます。人間の記憶だけに頼らない状態を作ります。

🔎

Codexが文脈を読める

Codexは、Vault内のLecture、PJメモ、テンプレート、AGENTS.mdを読んでから作業します。だから、依頼のたびに長い背景説明をしなくても、前提つきで動けます。

📝

作業結果が次回に残る

調査結果、判断材料、未解決のTODO、次アクションをObsidianに戻します。1回の作業を使い捨てにせず、次の作業の文脈として再利用します。

🧯 詰まりやすいところ

Codexプラグイン前提では、GitHub認証とローカルファイル権限が別物です。ここを先に潰すとセットアップが速くなります。

🔐

GitHub登録・ログイン

アカウント作成は github.com/signup。リポジトリ作成は github.com/new。公開設定は必ずprivate。

🔌

Codex GitHubプラグイン

GitHub連携はプラグイン接続で進める。Classic tokenを資料に貼る運用はしない。接続できていない時は、Codex側のプラグイン接続状態を先に確認する。

🗂️

フルディスクアクセス

CodexがDocuments配下のVaultを読めない、ファイル作成できない、Git操作で権限エラーが出る場合は、macOSのフルディスクアクセスにCodexを追加して再起動する。

🧭

原因の切り分け

GitHubプラグインの認証は「GitHub API用」。ローカルVaultを読む権限、ローカルgitの認証、GitHub Desktopの同期権限は別。どこで失敗しているかを分けて見る。

詰まり原因候補先に見る場所
Vaultが読めないmacOSのファイル権限不足フルディスクアクセス / Files and Folders
GitHubリポジトリ作成で止まるプラグイン未接続、権限不足CodexのGitHubプラグイン接続状態
pushで止まるremote未設定、ローカルgit認証未完了git remote -v / GitHub Desktop / SSH設定
Classic token時代より詰まる手動トークンは単純だが、権限分離が見えにくいプラグイン認証とローカル権限を分けて確認

🧪 実際にどう使うか

AIに慣れていない人は、まずこのまま真似してください。うまい依頼文を考えるより、型を使う方が早いです。

まず覚える依頼の型

  • 何をしたいかを1行で言う
  • 読んでほしい場所を指定する
  • 出してほしい形を指定する
  • 不明点は質問して、と添える

依頼テンプレ

  • 目的: 何を決めたい / 作りたい
  • 参照: 20_Context / 30_Projects / 70_Meetings
  • 出力: 要点 / 選択肢 / TODO / 共有文
  • 制約: 推測で埋めない / 不明点は質問

❓ よくある質問

AGENTS.mdはなぜ最初から入れる？

Codexの出力品質は、最初に読ませる前提で大きく変わります。配布版のAGENTS.mdを入れることで、人間が最初から同じ作業ルールでCodexを使えるようにします。

AXAD MCPのキーはこの資料に載せる？

載せません。キーや接続情報は別途安全な経路で共有します。この資料には、できること、使い方、注意点だけを書きます。

AXAD MCPは何に使う？

読み取り専用で、広告データの確認・集計・異常候補の抽出・レポート観点づくりに使います。この資料では「数字を見て、判断材料にする」使い方に絞ります。

エンジニア以外も使える？

使えます。基本は日本語でCodexに依頼します。ただし、Vault作成・権限・MCP接続など初回セットアップは管理者が支援する想定です。
